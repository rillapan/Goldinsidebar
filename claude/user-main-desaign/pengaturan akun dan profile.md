## Problem Statement

Halaman `/profile` saat ini hanya menampilkan info akun, membership, theme toggle, dan riwayat transaksi. Spec meminta satu halaman terpadu yang menggabungkan seluruh pengaturan user: identitas, tampilan (theme/bahasa/mata uang), notifikasi (saluran & jenis), trading defaults, dan bantuan — dengan navigasi yang intuitif di mobile.

## What Makes This Cool

User tidak perlu mencari-cari pengaturan di berbagai halaman. Satu halaman `/profile` mengandung semua yang mereka butuhkan untuk mengkustomisasi pengalaman platform mereka, dengan tab navigation yang responsif dan instant feedback (localStorage changes apply immediately without reload).

## Constraints

- Tidak ada state management eksternal baru — gunakan `useLocalStorage` pattern yang sudah ada untuk prefs tampilan
- Password change memerlukan satu endpoint baru di backend (`PATCH /api/auth/change-password`)
- Foto profil ditunda (hanya tampilkan initials avatar dari nama untuk sekarang)
- 2FA: tampilkan UI section dengan badge "Segera Hadir" — tidak diimplementasikan sprint ini
- Broker API: tampilkan placeholder "Hubungkan Broker (coming soon)" — tidak diimplementasikan sprint ini
- Notification channel preferences (WA/TG) disimpan localStorage saja — tidak perlu endpoint baru

## Premises

1. Notification preferences (toggle sinyal, toggle bias, saluran WA/TG/Browser) → localStorage ✅ dikonfirmasi user
2. 2FA → UI placeholder "Segera Hadir" sprint ini, implementasi penuh di sprint berikutnya
3. Broker API → placeholder section, tanpa implementasi nyata sprint ini
4. Format Mata Uang preference → localStorage; komponen yang menampilkan angka akan membaca preference ini
5. Password change → endpoint baru `PATCH /api/auth/change-password` di backend (2 field: currentPassword + newPassword)

## Approaches Considered

### Approach A: Tab Bar di Atas ✅ DIPILIH
  Summary: Tab bar horizontal di atas halaman dengan 5 tab. Satu route `/profile`, konten berganti per tab dengan animasi fade.
  Effort:  S
  Risk:    Low
  Pros:
    - Mobile-friendly: tab mudah di-tap, tidak perlu gestur slide
    - Tidak membutuhkan sidebar logic atau overlay state
    - URL tidak berubah → no hydration issue, no Suspense boundary baru
  Cons:
    - 5 tab cukup penuh di layar kecil (gunakan scroll horizontal atau icon+label pendek)
  Reuses: ThemeToggle, LanguageToggle yang sudah ada; glass-card pattern; useAuthStore

### Approach B: Hamburger Slide Panel
  Summary: Tombol hamburger membuka panel slide dari kanan dengan 4 section.
  Effort:  M
  Risk:    Medium
  Cons: Perlu overlay state, animation logic, dan trap focus untuk accessibility. Lebih rumit dari nilai yang didapat.

### Approach C: Sub-routes Per Section
  Summary: `/settings/display`, `/settings/notifications`, dll sebagai route terpisah.
  Effort:  L
  Risk:    Medium
  Cons: Overkill untuk feature yang bisa hidup dalam satu halaman. Tambah kompleksitas routing.

## Recommended Approach

**Approach A — Tab Bar** karena paling ringan, mobile-friendly, dan dapat diimplementasikan tanpa mengubah routing atau menambah state management baru.

## Backend Changes Required

### Existing endpoints (tidak berubah)
- `GET /api/profile` → user info + membership (sudah ada)
- `GET /api/payments/history` → riwayat transaksi (sudah ada, tampilkan 20 terbaru DESC)

### New endpoints needed

```
PATCH /api/auth/change-password
  Auth: verifyToken only (password change allowed for ALL authenticated users,
        termasuk expired members — security best practice)
  Body: { currentPassword: string, newPassword: string }
  Logic:
    1. Fetch user by req.user.userId
    2. bcrypt.compare(currentPassword, user.passwordHash) → if false → 400
    3. If newPassword === currentPassword → 400
    4. If newPassword.length < 8 → 400
    5. bcrypt.hash(newPassword, 12)  ← cost factor 12, sama dengan existing
    6. prisma.user.update({ passwordHash })
    7. Return { success: true } — JWT TIDAK direvoke, user tetap login
  Error codes:
    { error: "INVALID_CURRENT_PASSWORD" }  → 400
    { error: "PASSWORD_UNCHANGED" }         → 400
    { error: "PASSWORD_TOO_SHORT" }         → 400

PATCH /api/users/me
  Auth: verifyToken + checkMembership (hanya member aktif yang bisa update profil)
  Body: { name?: string, phone?: string }
  Validation:
    - name: min 2 char, max 100 char
    - phone: opsional, format Indonesia (08xx / 628xx), length 10-13 digit
      (validasi di frontend saja; backend hanya strip whitespace)
    - phone update: tidak ada side effect ke WhatsApp sender — stored as-is
    - phone tidak perlu unik di DB (tidak ada constraint unique di User.phone)
  Logic: prisma.user.update → return updated user object
  Response: { success: true, user: { id, name, phone, email, role, status,
              activeMembership: { startDate, endDate } } }
  Error codes (validation failures):
    { error: "NAME_TOO_SHORT" }  → 400 (if name present but < 2 char)
    { error: "PHONE_INVALID" }   → 400 (if phone present but bad format)
  Race condition: last-write-wins, idempotent — no DB versioning needed for name/phone updates
  Auth scope: verifyToken only (tidak pakai checkMembership — name/phone boleh diupdate
              oleh semua authenticated user, termasuk expired. Only password change uses verifyToken only too.)

KYC Status (Tab Akun):
  Untuk sprint ini: hardcode "Belum Terverifikasi" untuk semua user.
  Tidak ada field `isKyc` di Prisma User model sprint ini.
  Ketika field ditambahkan di sprint berikutnya: baca `user.isKyc ?? false`.
```

## Reviewer Concerns (unresolved, acceptable for sprint)

- `PATCH /api/auth/change-password` response is intentionally bare `{ success: true }` — no richer response needed for password change
- WA notification phone reads `user.phone` from DB at blast-time; updating phone via `PATCH /api/users/me` automatically takes effect on next WA blast — no code change needed in `notifications.ts`
- Expired users see the normal password change form; no UI warning needed (changing password is always allowed)
