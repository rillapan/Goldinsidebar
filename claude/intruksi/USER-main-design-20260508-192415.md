## Problem Statement

SINYAL COHIBA is a fully built SaaS platform (3-service monorepo: Next.js frontend, Express.js backend, Python FastAPI AI engine) for Indonesian XAUUSD retail traders. The system is functionally complete end-to-end: Twelve Data WS → Redis candles → pandas-ta + Claude API → signal → PostgreSQL + Socket.IO → Telegram/WA/Email.


This session was a code-level architecture and business logic validation to identify launch blockers before the first paying member is onboarded.

## Demand Evidence

- Market-level: Indonesian retail XAUUSD traders demonstrably pay Rp 100–500k/month for signal groups (signal provider market verified)
- Product-level: Pre-revenue; first paying member has not yet been acquired
- Core risk: Demand is unproven at the product level. The prior design session established this. The web platform was built before validation.

Target user currently: pays Rp 100–500k/month to join human-run signal groups on WhatsApp/Telegram, receives late signals (5–15 min after entry window closes), has no explanation for signals, and no accountability when signals are wrong. SINYAL COHIBA replaces this with AI-generated signals delivered instantly with reasoning.

## Architecture Findings

Five confirmed gaps from code review (agreed by founder in session):

### GAP 1 — False Performance Claim [CRITICAL — legal]
`frontend/src/app/page.tsx` contains **four** false statistics that must all be removed:

1. **Stats bar (~line 65):** "87%" win rate stat chip
2. **Stats bar (~line 66):** "1.240+" historical signals stat chip
3. **Stats bar (~line 67):** "850+ trader Indonesia" stat chip (also fictional — no users yet)
4. **FAQ (~line 33 of `steps`/FAQ array):** "Win rate rata-rata 87% berdasarkan data historis 1.240+ sinyal"

5. **`src/components/ui/pricing-section.tsx` (~line 134):** "87%" and "1.240+ sinyal historis" (win rate card in pricing panel)
6. **`src/components/ui/pricing-section.tsx` (~line 144):** "850+ trader" claim (same component)

Check all six locations. `pricing-section.tsx` is a separate component file — fixing only `page.tsx` leaves two live false claims.

Zero signals have been generated. Zero outcomes tracked. Publishing any of these on a financial signals product in Indonesia is false advertising and creates OJK regulatory exposure. All four must be replaced with honest copy (e.g., "Teknologi AI real-time", "Berbasis Claude AI", "Sinyal otomatis 24/5") before any user sees the landing page.

### GAP 2 — Telegram Invite Not Automated [CRITICAL — operational]
`notifications.ts:sendActivationNotification()` sends WA + email after payment confirms, but neither message contains a Telegram invite link. The Telegram Bot API `createChatInviteLink` is not called anywhere in the codebase.

**Impact:** A user pays via Xendit → webhook fires → membership activates → user receives "Membership Aktif" WA/email → lands on `/dashboard` → no way to reach the private Telegram channel without manually contacting the founder.

**Fix:** Add a `createTelegramInviteLink()` function to `notifications.ts` (alongside the existing `sendTelegram()`, reusing `TELEGRAM_BOT_TOKEN`). Call it inside `sendActivationNotification()`, include the link in both the WA message and email.

```
POST https://api.telegram.org/bot{TOKEN}/createChatInviteLink
body: { chat_id: TELEGRAM_CHANNEL_ID, member_limit: 1, expire_date: now + 172800 }
```

**Prerequisite:** The bot must be a **channel admin** with "Invite Users via Link" permission. Without this, the API returns `Bad Request: not enough rights`. Set this up in Telegram channel settings before writing any code.

**Placement in `sendActivationNotification()`:** Call `createTelegramInviteLink()` BEFORE `Promise.allSettled()`. Store the result in a variable, then pass the invite link into both the WA message and email HTML. If the invite call fails, set the invite link to an empty string and include the fallback text. The `Promise.allSettled()` call remains unchanged — invite link generation does not run inside it.

**Error handling:** If `createChatInviteLink` fails (Telegram API down, invalid token, bot not admin), log the error, send WA/email with fallback text "Hubungi admin @username untuk akses channel Telegram", and do NOT block the payment confirmation response.

**Invite expiry:** Links expire after 48 hours. If the user has not clicked within 48 hours, a new link must be generated manually (or via a `/refresh-telegram-invite` endpoint — defer to sprint 2). Add a note in the activation email: "Link ini berlaku 48 jam."

Required env var: `TELEGRAM_BOT_TOKEN` (already in use for broadcast), `TELEGRAM_CHANNEL_ID` (already present in `notifications.ts` line 14).

### GAP 3 — Expired Members Retain Channel Access [IMPORTANT — revenue]
The membership expiry cron (`cron.ts:0 * * * *`) deactivates memberships and sets `user.status = EXPIRED`. It does NOT call `kickChatMember` on the Telegram channel.

**Impact:** Every member who ever subscribed stays in the channel indefinitely. At 50 members this is a minor revenue leak. At 500 members it becomes a reason the business case breaks — paying members are subsidizing free access for churned users.

**Decision from session:** Defer to manual for now. At <20 members, manually kick expired users. Add automation when crossing 20 members.

### GAP 4 — Signal Outcome Tracking Not Built [IMPORTANT — included in Approach A]

> **Why this IMPORTANT gap is in Approach A while GAP 3 (also IMPORTANT) is deferred:** GAP 3 (expired members in channel) causes revenue leakage that scales with user count — at <20 members it costs 1-2 minutes/month of manual work. GAP 4 causes an empty performance history at day 1 for every user who logs into the dashboard. An empty history makes the product look broken, not just imperfect. It is a first-session trust issue, not a scaling issue.

### GAP 4 — Signal Outcome Tracking Not Built [IMPORTANT — included in Approach A]
`Signal` model has `status` (ACTIVE/TP_HIT/SL_HIT/etc), `exitPrice`, `pnlPips`, `closedAt` fields. Nothing updates them. Every signal remains ACTIVE forever.

**Impact:** The signals history page (once built) shows an infinite list of "ACTIVE" signals. The performance claim "AI lebih akurat dari grup signal manusia" cannot be substantiated. At renewal time, users cannot verify if the service delivered value.

**Fix:** Add Job 5 to `backend/src/lib/cron.ts` (currently 4 jobs). Run every 5 minutes:
1. Fetch all ACTIVE signals from PostgreSQL
2. Read current XAUUSD price from Redis `price:xauusd` (TTL 120s)
3. If `price:xauusd` is absent or timestamp is older than 120 seconds: skip cycle, log warning "price feed stale — skipping outcome check"
4. For each signal, apply:

| Signal type | TP_HIT condition | SL_HIT condition |
|---|---|---|
| BUY | `currentPrice >= takeProfit` | `currentPrice <= stopLoss` |
| SELL | `currentPrice <= takeProfit` | `currentPrice >= stopLoss` |

5. On match: update `status`, set `exitPrice = currentPrice`, `closedAt = now()`
   - Calculate `pnlPips`: for XAUUSD, 1 pip = $0.01 (the 4th decimal place in USD). Formula: `pnlPips = abs(exitPrice - entryPrice) / 0.01`. For a BUY that exits at TP: positive. For a BUY that exits at SL: negative. Store sign based on outcome (TP = positive for BUY/negative for SELL, SL = vice versa).
   - **Tiebreaker (TP and SL crossed in same cycle):** If `currentPrice >= takeProfit AND currentPrice <= stopLoss` (impossible in practice) or more realistically if price is between entry and both levels due to a price feed anomaly — prefer SL_HIT (conservative, trader-friendly). In practice this won't happen with valid signals.

**Sampling limitation (acknowledged):** 5-minute polling cannot detect intra-bar SL/TP crosses. If price spikes through the stop loss and recovers within 5 minutes, the cross is invisible. For pre-commercial trial (<10 members, no accountability expectation), this is acceptable. Accurate outcome tracking requires per-tick checking — defer until after first 5 paying members.

### GAP 5 — ATR Synthetic Proxy [KNOWN LIMITATION — deferred]
`signal_engine.py:76-83`: ATR calculated using `high = close * 1.001, low = close * 0.999`. For XAUUSD at $2,750, this produces an ATR proxy of ~$2.75 vs real M15 ATR of $3–$15 depending on volatility.

**Decision from session:** Replace with Twelve Data REST API M1 OHLC candles before commercial scale (>20 members). The tick-based proxy is acceptable for the first trial period. Twelve Data REST endpoint: `GET /time_series?symbol=XAU/USD&interval=1min&outputsize=200`.

## Additional Findings (not launch blockers, fix soon)

**No prompt caching on Claude API:**
Both `signal_engine.py` and `bias_engine.py` call Claude with a user message containing the full indicator data + instructions. The system-level instruction ("Kamu adalah AI Trading Analyst XAUUSD profesional...") is embedded in the user message, not in a `system` field. Adding a `system` field and enabling Anthropic prompt caching (`cache_control: {"type": "ephemeral"}`) on the system prompt would reduce Claude API costs by ~70% at high call frequency.

**No retry on daily bias cron:**
`cron.ts:Job 1` calls `POST ${AI_ENGINE_URL}/api/generate-bias` with a 60s timeout and no retry. If the AI engine is cold-starting or temporarily down at 07:00 WIB, that day's bias is lost. Add 1-2 retry attempts with 30s delay.

**checkMembership queries DB on every protected request:**
`auth.middleware.ts:checkMembership` hits PostgreSQL for every API call to premium routes. At <50 members this is fine. Add Redis caching (TTL 60s, key `membership:{userId}`) when crossing 50 members.

**Daily Bias direction vs Signal direction conflict:**
If bias_engine.py returns WAIT but signal_engine.py fires a BUY 30 minutes later, users receive contradictory guidance. No reconciliation logic exists. *Out of scope for Approach A — revisit in sprint 2 after first paying members. Do not implement yet.*

## Approaches Considered

### Approach A: Minimal Launch — Fix 3 Critical Blockers (SELECTED)

Fix the three issues that break core operations on day 1. Defer the rest.

- **Fix:** Remove false 87% / 1,240+ win rate claim from landing page
- **Fix:** Add `createChatInviteLink` call to `sendActivationNotification()` — automate Telegram onboarding
- **Fix:** Add signal outcome monitor cron job — update Signal status when price crosses SL/TP
- **Defer:** Telegram revocation on expiry (manual, <20 members)
- **Defer:** ATR proxy replacement (acceptable for first trial)
- **Defer:** Redis membership caching (add at 50+ members)

**Effort:** S — 3-5 days dev
**Risk:** Low

### Approach B: Solid Launch — Fix All Pre-Launch Blockers

Everything in A plus: automated Telegram revocation on expiry, Twelve Data REST OHLC integration, Redis membership status caching.

**Effort:** M — 1-2 weeks
**Risk:** Low, but delays first paying member

### Approach C: Back to Validation First

Shelve the web platform. Deploy only signal_engine.py + Telegram. Run 10-person trial. Return to platform after demand is proven.

**Effort:** S (already built)
**Risk:** Medium — abandons 3+ weeks of web platform work

## Recommended Approach

**Approach A.** Fix the three launch blockers, ship. At <20 members, manual Telegram revocation is 1-2 minutes/month. The bigger risk is velocity loss from over-engineering before first revenue. Approach B is the right long-term state — add it in sprint 2 after the first 5 paying members.
